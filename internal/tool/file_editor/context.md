# file_editor context文档

## 1. What this module is

`internal/tool/file_editor` 是一个面向大模型调用的本地文件操作模块。

它不是通用编辑器，不负责 UI，不负责版本控制，也不提供自由格式的编辑语义。

它的职责只有四个：

- 接收结构化文件操作命令；
- 校验参数与路径；
- 执行受控的本地文件读取或编辑；
- 返回稳定的 observation，供上层 agent 继续推理。

这个模块的目标是成为一个“可预测、可约束、可回滚”的文件工具层。

## 2. Current Status (NOW)

当前真实状态如下：

- 对外优先暴露 `FileEditor` 结构体实例，而不是包级执行函数；
- 默认实例可通过单例入口获取；
- 命令分发已收敛到 `FileEditor.Execute` 方法；
- 命令集合已固定为 `view`、`create`、`str_replace`、`insert`、`undo_edit`；
- `ValidateAction` 和 `ValidatePath` 已实现；
- 统一错误建模和 observation 结构已实现；
- `view` 已实现，且可用于文本、目录和图片查看；
- `create` 已实现最小闭环，可创建文件、写入内容并追加版本链记录；
- `str_replace` 已实现，支持正则唯一匹配、去除首尾空白后重试以及版本链记录；
- `fileCache` 已实现，可做落盘缓存、容量控制和 LRU 淘汰；
- `fileHistoryManager` 已实现，可做单文件历史快照管理；
- `insert` 已实现最小闭环，支持按行插入、版本链记录和上下文返回；
- `undo_edit` 已实现最小闭环，可消费最近历史版本并写回文件；
- 文本展示已开始收敛到统一 helper，负责描述、截断和行号输出。

对大模型而言，这意味着：

- 这个模块已经有清晰协议；
- 查看链路已可用；
- 编辑基础设施已准备好；
- `create`、`str_replace`、`insert` 和 `undo_edit` 已可用。

## 3. Priority Order

当前优先级顺序如下：

1. 保持命令协议稳定。
2. 保持 `view` 行为稳定，因为它是编辑前确认上下文的前置能力。
3. 通过 `FileEditor` 实例统一持有 `historyManager`、大小限制和运行时上下文。
4. 收敛 `undo_edit` 的最终回滚语义。
5. 收敛 `create` 的最终撤销语义。
6. 收敛 `str_replace` 的正则边界与消息格式。
7. 收敛 `insert` 的插入语义和上下文窗口规则。

如果必须做取舍，优先保证“正确和可回滚”，不要优先追求“更强大但更模糊的编辑能力”。

## 4. Hard Constraints (DO / DON'T)

### DO

- 只接受绝对路径。
- 所有命令都先经过 `ValidateAction` 和 `ValidatePath`。
- 所有子命令优先实现为 `FileEditor` 方法，而不是继续扩散纯函数式入口。
- 默认调用路径优先走 `FileEditor` 单例实例。
- 编辑命令返回统一的 `FileEditorObservation`。
- 编辑已有文件内容前，优先保存旧内容到 history。
- 将错误表达为稳定、可读、可复用的错误 observation。
- 对文本、目录、图片和二进制文件继续保持明确边界。
- 优先做确定性行为，不做模糊推断。

### DON'T

- 不要接受相对路径。
- 不要重新引入包级 `Execute` 这类函数式包装作为主要入口。
- 不要让子命令继续直接依赖包级临时状态。
- 不要把这个模块做成自由文本 patch 引擎。
- 不要在匹配不唯一时执行替换。
- 不要在越界行号上执行插入。
- 不要对二进制文件做文本编辑。
- 不要绕开 `fileCache` / `fileHistoryManager` 自己保存撤销状态。
- 不要把 `undo_edit` 设计成 Git 替代品。
- 不要为了“更智能”而放宽当前安全边界。

## 6. Command Contract

### `view`

- 当前已实现。
- 允许查看普通文本文件。
- 允许按 `view_range` 查看指定行范围。
- 输出带行号。
- 允许查看目录，但只展开两层深度。
- 默认过滤隐藏文件和隐藏目录。
- 允许查看支持的图片文件，并返回 image content。
- 拒绝二进制文件。
- 对输入文件大小和输出内容大小有上限。

### `create`

- 当前已实现最小闭环。
- 目标文件应当不存在。
- 父目录应当已存在。
- 内容来自 `file_text`。
- 创建成功后会追加一条版本链记录。
- 成功后应返回 `prev_exist=false`。
- 当前只要求最小创建闭环，不要求复杂创建策略。

### `str_replace`

- 当前已实现。
- 目标文件必须存在。
- 目标文件必须是可识别文本文件。
- 先按正则表达式匹配 `old_str`。
- 如果首次没有匹配，再对 `old_str` 和 `new_str` 去除首尾空白后重试一次。
- 最终必须恰好匹配一次。
- 0 次匹配报错，并明确说明没有进行替换。
- 多次匹配报错，并明确说明没有进行替换。
- 成功替换前应把旧内容追加到版本链。
- 成功后应返回旧内容和新内容。
- 成功消息必须返回受影响行预览，并包含“检查一下看是否符合预期，否则可以重新编辑”。

### `insert`

- 当前已实现最小闭环。
- 目标文件必须存在。
- 目标文件必须是可识别文本文件。
- `insert_line` 从 1 开始。
- `insert_line` 必须位于 `[1, 当前最大可插入行号]` 范围内。
- 空文件允许在第 1 行插入。
- 插入前应把旧内容追加到版本链。
- 写入采用临时文件原子替换原文件。
- 成功后应返回旧内容和新内容。
- 成功消息必须明确指出修改的是哪个文件。
- 成功消息必须返回插入位置附近最多 5 行上下文。

### `undo_edit`

- 当前已实现最小闭环。
- 应只回滚最近一次可撤销编辑。
- 应从 `fileHistoryManager` 中取回最近快照。
- history 为空时应返回明确失败结果。
- 文件被外部删除或不可恢复时应返回稳定错误。
- 成功消息必须返回写回后的内容预览。
- 它只解决最近编辑回滚，不承担长期版本管理职责。

## 7. Definition of Done

下面的“完成”指的是功能达到当前阶段可接受的最小闭环，不要求一步做到最复杂版本。

### `view` 算完成，当且仅当：

- 能稳定查看普通文本文件，并支持 `view_range`。
- 能稳定查看目录，并保持两层深度和隐藏项过滤。
- 能稳定查看受支持图片，并返回 image content。
- 会拒绝二进制文件、超大文件和超大输出。
- 返回格式仍然是统一的 observation，而不是命令私有输出。

### `create` 算完成，当且仅当：

- 能创建不存在的目标文件。
- 能把 `file_text` 正确写入文件。
- 能把这次操作加入 undo log，使后续 `undo_edit` 有机会处理它。
- 能返回正确的 `FileEditorObservation`。
- 对“文件已存在”或“父目录不存在”等情况返回稳定错误，而不是 panic。

### `str_replace` 算完成，当且仅当：

- 能读取目标文本文件并定位 `old_str`。
- 先按正则匹配；首次未命中时，会在去除 `old_str` 和 `new_str` 首尾空白后重试一次。
- 只在唯一匹配时执行替换。
- 替换前会把旧内容写入 history / undo log。
- 替换后会把新内容正确写回文件。
- 会返回包含旧内容和新内容的 observation。
- 对 0 次匹配、多次匹配、非文本文件、目标文件不存在等情况返回稳定错误。
- 成功 observation 文本中包含受影响行预览和“检查一下看是否符合预期，否则可以重新编辑”。

### `insert` 算完成，当且仅当：

- 能按 `insert_line` 在文本文件中插入内容。
- 插入前会把旧内容写入版本链。
- 插入后会把新内容正确写回文件。
- 能处理空文件第 1 行插入。
- 会返回包含旧内容和新内容的 observation。
- 成功 observation 文本中包含目标文件路径和插入位置附近最多 5 行上下文。
- 对越界行号、非文本文件、目标文件不存在等情况返回稳定错误。

### `undo_edit` 算完成，当且仅当：

- 能从 `fileHistoryManager` 取到最近一次可撤销快照。
- 能把目标文件恢复到上一个状态。
- 恢复后会返回撤销前内容和撤销后内容。
- history 被成功消费，不会无限重复撤销同一个版本。
- 成功 observation 文本中包含目标文件路径和写回后的内容预览。
- 对无历史、文件已被外部删除、历史损坏等情况返回稳定错误。

### `fileCache` 算完成，当且仅当：

- 支持 `get` / `set` / `delete` / `clear`。
- 具备大小限制和 LRU 淘汰。
- 采用原子写入，避免明显的部分写入损坏。
- 重启后仍能从磁盘恢复当前缓存状态。

### `fileHistoryManager` 算完成，当且仅当：

- 能按文件维度保存多个历史版本。
- `add` 生效，`pop` 遵循后进先出。
- 支持最大历史条数裁剪。
- metadata 与版本缓存不一致时能做最基本修复。
- 支持 `delete` 和 `clear`。

## 8. Known Risks

- `FileEditor.cwd` 和大小字段已经进入结构体，但后续命令还需要继续真正消费这些上下文。
- `fileHistoryManager` 已就绪，但何时写入 history 还需要在编辑命令里保持一致。
- `create` 已写入 undo log，但 `undo_edit` 对“恢复为空文件”还是“删除新建文件”的最终语义仍未确定。
- `str_replace` 已使用正则唯一匹配，后续需要继续收敛复杂正则模式下的替换语义。
- `insert` 已实现，但“插入到某一行”是否统一解释为“在该行前插入”还需要继续收敛为最终约定。
- 展示 helper 已统一基础文本预览，但不同命令的描述头文案仍需要继续收敛。
- 并发编辑不是当前设计重点，多 agent 或外部进程同时写文件时可能产生状态竞争。
- 当前设计默认更适合文本文件，复杂编码和大体积文件的编辑策略尚未明确。
- 如果未来放宽替换或插入规则，容易降低可预测性并增加误改风险。

## 9. Next Step

下一步按这个顺序推进：

1. 视 `undo_edit` 的实现结果，决定 `create` 的最终回滚语义。
2. 收敛 `str_replace` 在复杂正则场景下的约束与测试。
3. 收敛 `insert` 在不同换行风格和空插入文本场景下的约束与测试。
4. 收敛 `undo_edit` 在多次连续回滚场景下的约束与测试。
5. 继续统一各命令的展示文案与预览策略。
6. 完成后再评估是否需要更强的目录约束、权限控制或并发保护。
